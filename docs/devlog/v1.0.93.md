## 1. 项目概览

**job-pro** 是一个 TypeScript CLI，可直接对接 50 家中国大厂的公开招聘 API，无需登录、无需代理服务器。

**核心功能：**

| 命令 | 说明 |
|---|---|
| `search <keyword>` | 搜索岗位 |
| `detail <post_id>` | 获取 JD 详情 |
| `all` | 拉取全量岗位 |
| `match <resume>` | 简历匹配，返回最相关岗位 |
| `resume-check` | 简历结构检查 |
| `apply <post_id>` | 自动投递（Phase 2） |
| `find <keyword>` | 跨 50 家并行搜索 |
| `notices` / `flow` | 招聘公告 / 问答 |
| `memory` | 本地求职进度记录 |
| `profile` | 个人信息配置（用于自动投递） |

**适配器架构：** 每家公司一个 adapter，实现统一的 `CompanyAdapter` 接口（9 个动词）。  
**ATS 家族：** Bespoke (23)、Feishu (9)、Moka (7)、Beisen Wecruit (2)、Beisen iTalent (2)、Greenhouse/Lever (3)、Liepin (4)。

## 2. 已修复 — 本次 Session（2026-05-19）

本次 session 聚焦于 `matchResume` 命令在多家公司返回 0 matches 的问题，以及相关代码 bug。

### 2.1 根本原因：AND 关键词搜索导致结果池为空

**文件：** `tencent.ts`（已有正确实现，其余均为旧版）  
**症状：** bytedance、alibaba、netease、xiaomi、didi、zhipu、bilibili、jd、liauto 等大量公司 `match` 返回 0 条。  
**原因：** `matchResume` 用 `terms.slice(0,3).join(" ")` 拼成一个 AND 查询（e.g. `"pytorch llm rag"`），各家 API 将其视为所有词必须同时出现，结果池几乎必然为空。  
**修复：** 改为 fan-out 策略——`pickDistinctiveTerms` 选出有区分度的词，每个词单独并行搜索，按 `post_id` 去重合并，再整体评分；全失败时退回无关键词 broad search。

**修改的文件：**
- `tencent.ts` — 将 `pickDistinctiveTerms` 改为 `export`，供其他 adapter 引用
- `bytedance.ts` — fan-out（双路：searchPositions + `/search/job/posts`）
- `xiaomi.ts` — fan-out（双路，保留 `channel` 参数）
- `zhipu.ts` — fan-out（双路，无 `recruitment_id_list`）
- `bilibili.ts` — fan-out（单路 searchPositions）
- `didi.ts` — fan-out（单路）
- `iflytek.ts` — fan-out（单路）
- `wecruit.ts` — fan-out（单路，factory 模式，覆盖 sensetime + horizonrobotics）
- `alibaba.ts` — fan-out（单路）
- `netease.ts` — fan-out（单路）
- `jd.ts` — fan-out（单路）
- `feishu.ts` — fan-out（双路，factory 模式，覆盖 minimax/nio/baichuan/zerooneai）
- `liauto.ts` — fan-out（单路）
- `agibot.ts`, `vivo.ts`（间接引用修复） — 同步修改 import

### 2.2 `fetchAllPositions` 类公司无 fallback

**文件：** `vivo.ts`、`byd.ts`、`weibo.ts`、`pdd.ts`  
**症状：** 上述公司 `match` 返回 ok: true 但 matches 为空。  
**原因：** 这类公司不走关键词搜索，而是 `fetchAllPositions` 拉全量数据后本地评分。当简历关键词全为英文（如 pytorch/llm）而岗位标题全为中文时，所有 `score = 0`，没有 fallback 直接返回空数组。  
**修复：** `scored` 为空时，改为返回 `all.positions.slice(0, topN)`（不过滤，按原始顺序兜底）。

### 2.3 JD 的 pageSize 限制

**文件：** `jd.ts`  
**症状：** `jd match` 返回 `"no positions returned"`，即使无关键词 broad search 也失败。  
**原因：** JD 的 `campus.jd.com` API 对 `pageSize > 20` 静默返回空列表（`total: 23, positions: []`）。原代码用 `pageSize: 200`。  
**修复：** 改为 `pageSize: 20`（keyword search + broad fallback 均如此）。

### 2.4 pingan 拒绝英文关键词

**文件：** `pingan.ts`  
**症状：** 所有英文关键词（包括单词如 `python`）均返回 `ok: false`，消息为"请求成功"（误导性）。  
**修复：** 去除关键词搜索，直接使用无 keyword 的 `searchPositions({ pageSize: 100 })` broad fetch，再本地评分。

### 2.5 baidu pageSize 限制

**文件：** `baidu.ts`  
**症状：** `Illegal argument : pageSize` API 报错。  
**原因：** baidu API 带关键词时 `pageSize` 上限约为 20，原代码用 `pageSize: 100`。  
**修复：** 改为 `pageSize: 20`。

### 2.6 liauto `keyword` 变量未定义崩溃

**文件：** `liauto.ts`  
**症状：** `liauto match` 抛出 `Error: keyword is not defined`。  
**原因：** 应用 fan-out 修复时删除了 `const keyword = ...`，但返回对象里遗留了 `keyword_used: keyword` 字段。  
**修复：** 删除 return 语句中的 `keyword_used` 字段。

## 3. 已修复 — 历史版本（v1.0 系列）

### v1.0.93 — match 命令端到端可用（发布前最后一版）

- **简历输入格式扩展：** 新增 `--resume <path>` flag，支持 `.pdf`（pdf-parse + pdftotext 双解析器取长者）、`.docx`（mammoth）、`.json`（HA7CH/jsonresume 格式）、`.txt/.md`（原有）；`profile.resume_path` 作兜底。
- **term 误提取修复（false positive）：** `termMatches` 原用 bare `includes`，导致 `rust` 匹配 "Trustworthy"、`lua` 匹配 "evaluation" 等。改为全部使用 word-boundary regex（`(^|[^a-z0-9])term(?![a-z0-9])`），中文词保持 substring 匹配。
- **tencent fan-out 策略：** 新增 `GENERIC_SEARCH_TERMS` 过滤通用词（python/docker/linux 等），用 `pickDistinctiveTerms` 选有区分度的词单独搜索。**注意：当时仅对 tencent 生效，其余 adapter 未同步（本次 session 已补齐）。**
- **学历过滤（degree-aware sort）：** `matchResume` 接受 `userDegree`，识别 JD 中的学历要求（本科及以上/硕士/博士），返回结果添加 `meets_degree_requirement` 标注，不符合的排在后面不过滤。
- **测试覆盖：** `pnpm test:match`（21 条断言，覆盖 word-boundary、fan-out recall、学历检测）。

### v1.0.88–1.0.92 — CDP 自动投递

- **DOM 驱动投递：** `--via-cdp` flag 启用 Puppeteer CDP walker，直接在浏览器中填写表单并提交，覆盖无法直接拦截 API 的公司。
- **原生 `<select>` 支持：** Greenhouse 表单中的多选下拉框（work-auth、sponsorship 等）之前被 `page.type()` 跳过，现已正确处理。
- **`--debug-submit-to`：** 将 POST 请求转发到 echo server（如 httpbin）验证 wire format，不影响上游。

### v1.0.73–v1.0.87 — 覆盖率与验证

- 45/50 adapter 完成 apply endpoint 验证（`endpoint_verified: true`）。
- 5 家结构上无法 API 投递（见下文"未修复"）。
- `JOB_PRO_ALLOW_SPECULATIVE_ENDPOINT=yes` 第 4 道安全门，防止误投未验证 endpoint。

### v1.0.50 以前 — 基础建设

## 4. 已知未修复的问题

### 4.1 lilith 需要 puppeteer-core + Chrome

- **影响：** `lilith match`、`lilith apply` 完全无法使用。
- **原因：** Lilith 使用字节跳动反爬机制（`portal_type=6`），每次请求需要浏览器 mint 一个 `_signature` token，必须通过 Puppeteer CDP 获取。
- **解决方案：** `npm install puppeteer-core`，并安装 Google Chrome（或设置 `$JOB_PRO_CHROME=/path/to/chrome`）。
- **文件：** `cli/src/cdp.ts`、`cli/src/feishu.ts`（lilith 分支）。

### 4.2 antgroup 频繁限流

- **影响：** `antgroup match/search` 偶发 `系统繁忙，请稍后重试`。
- **原因：** `talent.antgroup.com` API 有较严格的频率限制，在并行测试或短时间内多次调用时触发。
- **解决方案：** 无代码修复方案，需要加 retry + 随机延迟，或错峰调用。
- **文件：** `cli/src/antgroup.ts`。

### 4.3 huawei match 慢（>30 秒）

- **影响：** 在并行批量测试时超时失败，但单独运行正常（约 40–60s 返回 5 matches）。
- **原因：** Huawei 的适配器对每个候选岗位都单独调用 `fetchPositionDetail`（串行），而非在 search 结果中直接评分。
- **解决方案：** 在本地评分阶段先用 search 结果中的字段（title/department）筛选，只对 top 候选做 detail 拉取（并行化 + 提前截断）。
- **文件：** `cli/src/huawei.ts`。

### 4.4 BYD / Weibo fallback 返回不相关岗位

- **影响：** `byd match`、`weibo match` 返回 5 条，但岗位与 AI/ML 简历无关（BYD 会出现设备工程师、技师等）。
- **原因：** 这两家公司的校招岗位主体是制造/汽车方向，AI 岗极少；加上 fallback 按原始顺序返回而非按相关性，导致质量差。
- **解决方案：** fallback 阶段改为优先按 title 关键词做初步过滤，或在 `extractResumeSignals` 中生成中文同义词（如 llm→大模型、rag→检索增强）再匹配。
- **文件：** `cli/src/byd.ts`、`cli/src/weibo.ts`。

### 4.5 extracted_terms 含大量非技术词

- **影响：** `match` 的 `extracted_terms` 里混入 "Curriculum"、"Vitae"、"PERSONAL"、"SUMMARY"、"years" 等英文简历排版词，拉低搜索精度。
- **原因：** `extractResumeSignals` 的词表过滤不够，`GENERIC_SEARCH_TERMS` 只过滤通用技术词（python/docker 等），没过滤非技术词。
- **解决方案：** 在 `tencent.ts` 的 `GENERIC_SEARCH_TERMS` 或专门的 stopword 列表里添加简历排版常见词；或用正则限定只提取看起来像技术词的 token（全小写、长度 ≥ 4、非纯字母的组合词等）。
- **文件：** `cli/src/tencent.ts`（`extractResumeSignals` 函数）。

### 4.6 5 家公司结构上无法 API 自动投递

- **影响：** `apply` 命令对以下 5 家返回 `submit_kind: "external"`，不能自动提交。
- **公司：** hikvision、cicc、cainiao、webank（均通过猎聘 recruiter chat）、unitree（微信小程序 QR 码）。
- **原因：** 这些公司的投递流程走 IM 渠道或微信，无公开 HTTP API。
- **解决方案：** 结构上无解，只能提示用户手动操作。

### 4.7 JD 只能获取 3 个岗位

- **影响：** `jd match` 候选池只有 3 条（`campus.jd.com` 当前实际开放岗位极少）。
- **原因：** JD 校招 API 显示 `total: 23` 但 page 1 只返回 3 条，page 2 返回 0。可能是季节性（非招聘季岗位未公开），也可能是 API 行为。
- **解决方案：** 待招聘季再测试；或调查 JD 的其他招聘入口（社招 API `hr.jd.com`）。

## 5. 预期目标 / 功能路线图

### 5.1 【高优先级】match 质量提升

- [ ] **中文同义词扩展：** `extractResumeSignals` 生成中文技术同义词映射（`llm → 大模型`、`rag → 检索增强`、`reinforcement learning → 强化学习`），大幅提升对中文 JD 的命中率。
- [ ] **Stopword 过滤：** 过滤简历排版词（见 4.5），减少 terms 噪声。
- [ ] **跨 adapter 学历过滤同步：** 学历过滤目前只对 tencent 完整实现，其余 adapter 应同步（`userDegree` 参数 + `degree_required` 字段）。

### 5.2 【高优先级】自动投递完善

- [ ] **30 家 🔑 session-required adapter 实地验证：** 目前 `endpoint_verified: true` 的有 45 家，但多数是 anon-probe（HTTP 探针）而非真正的端到端投递测试。需用真实 session cookie 对每家公司实际提交一次测试申请。
- [ ] **Session 捕获流程文档化：** 补充各 ATS 家族（Feishu/Moka/Wecruit/bespoke）的 session export 操作步骤截图。
- [ ] **Moka `fetchPositionDetail` 实现：** `moka.ts` 的 detail 接口目前返回"not implemented"（见 `moka.ts:445`），因为需要同样的 AES 加密 session 流。

### 5.3 【中优先级】性能优化

- [ ] **huawei match 并行化：** 先批量评分 title，再对 top-N 并行拉 detail（见 4.3）。
- [ ] **分页完整性：** 部分公司（jd、bilibili 等）的 `fetchAllPositions` / `searchPositions` 只拉了第一页，应完整分页直到 total 耗尽。

### 5.4 【中优先级】新公司覆盖

- [ ] 当前覆盖 50 家，可考虑扩展：商汤、旷视（独立入口）、理想汽车（已有但季节性失效）、字节跳动海外岗位（不同 portal）。

### 5.5 【低优先级】体验改善

- [ ] **`find` 命令结果质量：** 目前 `find` 只做关键词搜索，可加入 resume match 模式（`find --match resume.pdf`），跨 50 家并行返回 match 结果。
- [ ] **`memory` 与 apply 联动：** 投递成功后自动写入 `memory event applied`。
- [ ] **match 结果缓存：** 短时间内重复调用同一公司的 match 时可缓存搜索结果（岗位信息 TTL ~1h）。

## 6. 快速上手验证

```bash
# 进入 CLI 目录

# 批量测试 49 家（需要 Node 18+）
```

**批量测试脚本** `/tmp/match_all.mjs`（替换 RESUME 路径后可用）：

```js
import { execFile } from "node:child_process";
import { promisify } from "node:util";
const exec = promisify(execFile);
const RESUME = "/path/to/resume.pdf";
const CLI_DIR = "/path/to/job-pro/cli";
const TSX = "~/.npm/_npx/fd45a72a545557e9/node_modules/.bin/tsx"; // 或 npx tsx
const COMPANIES = [
  "bytedance","alibaba","meituan","xiaohongshu","jd","kuaishou","xiaomi",
  "baidu","netease","didi","bilibili","pdd","nio","minimax","huawei","weibo",
  "mihoyo","pingan","sensetime","trip","unitree","byd","antgroup","liauto",
  "moonshot","zhipu","hikvision","iqiyi","megvii","lilith","agibot","deepseek",
  "zerooneai","galaxyuniversal","stepfun","cicc","baichuan","xpeng","weride",
  "hoyoverse","iflytek","oppo","vivo","sf","cainiao","geely","webank",
  "horizonrobotics","cambricon"
];
const results = await Promise.all(COMPANIES.map(async (company) => {
  try {
    const { stdout } = await exec(TSX, ["src/index.ts", company, "match", RESUME],
      { cwd: CLI_DIR, timeout: 30000 });
    const d = JSON.parse(stdout);
    return { company, ok: d.ok, matches: (d.matches||[]).length };
  } catch(e) { return { company, ok: false, matches: 0 }; }
}));
const ok = results.filter(r => r.ok).length;
console.table(results);
console.log(`${ok}/${results.length} OK`);
```
